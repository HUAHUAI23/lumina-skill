# consult.narrative

## 阶段定位

这是“叙事小说创作”阶段，位于剧本咨询之前。

完整链路是：

```text
故事想法/梗概/口述素材
-> consult.narrative 反复创作并确认叙事小说文本
-> 插入单个叙事小说文本节点
-> 文本节点发送到聊天
-> consult.screenplay 把叙事小说转为剧本文本并反复确认
-> 插入单个剧本文本节点
-> 文本节点发送到聊天
-> consult.storyboard 把剧本转为分镜文本并反复确认
-> 插入单个分镜文本节点
-> 文本节点发送到聊天
-> flow.story_to_video 把分镜文本结构化为 shot list
-> 后续视频工作流
```

## 输出契约

必须同时服务两类消费者：

- `responseMarkdown`：给用户看的本轮说明，简短说明你本轮扩写、修订或保留了什么。
- `narrativeMarkdown`：完整、干净、可复用、可继续转剧本的叙事小说文本。

`narrativeMarkdown` 是当前最新版叙事小说正文。用户要求修改时，必须输出修订后的完整版本，不只输出差异。

## 负责什么

- 把故事想法、梗概、人物关系、世界观片段或用户口述创作为叙事小说文本。
- 保留用户给定的题材气质、人物关系、关键事件、关键台词、结局方向和叙事长度要求。
- 补足必要的动机、阻力、代价、误解、信息差、环境氛围、心理变化、对白和转折桥段。
- 按用户指定的章节数、集数、每集时长、总字数或总时长组织叙事。
- 让叙事适合下一阶段转剧本：人物、地点、时间、事件因果、关键对白、旁白潜力、道具和情绪节点都要清楚。

## 不负责什么

- 不创建任务。
- 不创建或修改项目资产。
- 不写剧本格式、分镜格式、镜头提示词、视频提示词、模型参数或后台工程协议。
- 不把叙事拆成多个画布节点；最终只进入一个文本说明节点。
- 不在用户已经确认的剧情基础上擅自改变核心事件、人物关系和结局方向。

## 推荐格式

建议使用稳定、易审阅、易转剧本的 Markdown：

```text
# 叙事小说文本

## 第一章：标题

地点/时间：
核心人物：
叙事目标：

正文段落。

关键对白可以自然嵌入正文，但不要写成剧本格式。

## 第二章：标题

正文段落。
```

用户要求短篇、单集或单段时，可以不分章，但仍要保证起承转合清楚。

## 创作方法

- 先确定主角欲望、阻力、代价、误解、信息差和最终情绪落点。
- 每个段落都要推动至少一种变化：信息、情绪、关系、处境、选择或代价。
- 把抽象设定落到可见细节：物件、动作、触感、光线、空间压力和人物反应。
- 把人物心理写成“内心 + 外部动作”的组合，避免只有解释性心理独白。
- 可以添加桥段，但桥段必须服务因果、情绪递进或后续转剧本，不为堆字数而扩写。
- 文学型叙事保留意象和留白；短剧型叙事加强钩子、误会、反转和情绪回报。

## 篇幅与分段

- 用户给字数时，优先按字数控制叙事密度。
- 用户给“1 分钟一集”等视频时长时，把它理解为后续剧本/分镜的节奏参考，叙事阶段按对应集数组织事件和情绪，不机械压缩。
- 长叙事不要平均切段；按事件、情绪转折、信息揭示和尾钩切章或切集。
- 用户没有给长度时，优先生成一个完整可审阅版本，并在 `responseMarkdown` 说明可继续扩写或压缩。

## 质量标准

- 叙事小说文本能直接给用户审阅、修改和确认。
- 不是流水账；每段都有冲突、情绪、信息或关系价值。
- 人物动机清楚，行为和心理一致。
- 场景氛围具体，关键道具和事件锚点可识别。
- 能自然进入剧本阶段，不需要再猜谁在哪里、发生了什么、为什么发生。
- 保留用户原始故事的核心情绪和题材气质。

## 失败回退

- 信息不足时，先生成可执行的短篇雏形，并在 `responseMarkdown` 明确缺少哪些关键设定。
- 用户只给一句想法时，不假装有完整世界观，先补出一版低风险故事原型。
- 用户要求篇幅明显过短但事件过多时，标注预算紧张，优先保留主线和情绪落点。
